最全的Android进程间通信

发表于 2017-03-11 | 分类于 进程 |

从官网文档出发，结合实例，给出最全的进程间通信方案。讨论如何多进程通信，以及各个方案的优缺点。

问题

  1. Android进程间通信的方式？
  2. Android进程通信的详细流程？
  3. 如何搭配Service完成进程通信工作？

开刀

Android进程通信的方式

  • 同步和异步远程过程调用(RPC)
  • 通过Messenger进行消息通信
  • 通过ResultReceiver返回数据

进程通信的详细流程

引入概念

Android RPC

IPC由LinuxOS管理，通常支持一些IPC的技术，比如说信号，管道，消息队列，信号量，共享内存等。而在Android定制化的Linux内核里，Linux的IPC技术被替换为了Binder框架(进程间的一个RPC机制)。

主要的步骤：

  • 方法和数据分解，也叫封送
  • 发送编组过的数据信息到远端进程
  • 在远端进程中改组数据信息，也叫做解封送
  • 发送返回值给原始进程
Binder

Binder允许程序在运行于不同进程的线程间发送方法和数据。服务端进程定义了一个由android.os.Binder类支持的远端接口，然后在客户端进程中的线程可以通过Binder访问这个远端接口。

先来看下通过binder实现的IPC过程：

默认情况下客户端线程调用transact()会被阻塞，直到在远端线程中onTransact()方法执行完毕。IPC过程中传输的数据包含了Parcel对象，根据官方文档，Parcel是一个数据和对象引用的容器，在IPC过程中比Serializable更高效。另外onTransact()方法执行在取自binder线程池的一个binder线程中，这个线程池只会处理来自其他进程进来的请求，一般16个线程容量，版本不同可能会有差异。换句话说，可以在每个进程中并发处理16个远程调用。

IPC过程是双向的，图中的流程会反转，所以异步的IPC会相当重要。

如果服务端进程开始了一个传输过程，客户端进程在执行onTransact()方法时，调用transact()发送一个请求到服务端进程，客户端进程将不会在binder线程中接收进来的请求，而是等待第一次传输过程完成。

为了解决这个场景，Binder也支持异步，通过设置IBinder.FLAG_ONEWAY。onway关键字可以在接口上，也可以在接口的方法上。接口上，接口所有方法异步；方法上，仅这个方法异步。设置这个标志后，客户端线程调用transact()不会阻塞且立刻返回，Binder会继续在服务端进程的binder线程中调用onTransact()方法，但并不会同步返回结果给客户端线程。

Messager

Android平台提供了一个高效的通过消息传递的内部线程通信方式。然而，它需要线程运行在同一进程中，因为Message对象被线程分配在共享内存里。如果线程运行在不同的进程里，它们没有共享消息的任何共享内存。相反，消息必须使用Binder框架跨进程通信。这个时候，我们可以考虑android.os.Messenage这个类，发消息到远端进程中线程绑定的Handler。

Messenger 利用了Binder框架同时发送Message对象和Messenger的引用到客户端进程，Handler不会跨进程发送，而Messenger作为了中间者。

ResultReceiver

ResultReceiver主要用于连接组件进行通信，比如说一个Activity启动了一个Service时传入ResultReceiver，在Service中操作完成后，可以通过ResultReceiver发送结果回Activity。它内部原理，也利用了Binder框架，只是设计有一定缺陷。

这个通信过程仅仅限于同一个包下，也就说本应用中。如果要跨进程通信，需要做点手脚。有两个方案：

  • 在传入ResultReceiver给Services时，对ResultReceiver进行包装，使得它适合跨进程通信。
  • Service中接收这个ResultReceiver时，重新设置其类加载器，使得能够获取ResultReceiver，这一点，在官方文档中有提到。

搭配Service完成进程通信工作

Service是四大组件之一，它可以后台运行。这个后台的意思，只是无前台UI元素，Service还是默认运行在当前进程的主线程里。所以不能做过多长时任务操作，比如说网络下载，同样是需要另起线程以避免阻塞住线程，引起ANR。Service是连接组件的纽带，在同一进程中可以被其他组件访问，也可以暴露给远端进程访问。

有人说，有了线程，还要Service干嘛。这主要是功能角色不同，主要区别如下：

  • 线程是进程工作的执行者，而在Activity中如果新启一个线程，如果Activity结束了，而线程还在，无法在Activity再次启动后对这个线程进行掌控。失控的线程如果有UI元素的引用，是一个危险事情。而Service恰恰就能解决这个问题，它可以在任何地方，再次启动或者绑定。
  • Service无UI元素，可以用来暴露给其他进程进行访问，在多进程的场景中，可以更好衔接工作，比如说推送等。Service里可以搭配线程，进行耗时任务，比如说IntentService。

Service启动方式

在Android5.0之后，官方建议显式启动Service，即明确指定类名或包名启动Service。两个启动方式：

  • startService
  • bindService

两种方式最主要的区别是：binService可以调用Service的方法以及生命周期的流程不同。

IPC通信的步骤

  1. 在服务端，新建AIDL接口并编译。
  2. 查看路径(build/generated/source/aidl/debug/包名) 下是否生成包含包装了Binder框架的接口。
  3. 创建服务端Stub的方法实现类。
  4. 新建服务，并将Stub实现类作为IBinder返回。
  5. Manifest文件中，配置服务，并设置 android:enabled=”true”，暴露服务给远端进程。

总结

在一个应用中大多数进程间通信被幕后的高级组件掌控，但如果有需要，我们可以自己实现Binder机制：RPC和Messenger。如果需要并发处理到来的请求，可以考虑RPC。反之，我们可以考虑Messenger简单地实现通信，记住Messenger是单线程运行的。而ResultReceiver默认在同一个进程下可以搞事，进程间通信还得包装一下，可以不考虑，它内部也是Binder机制，不过比较鸡肋罢了，还不如用RPC。
